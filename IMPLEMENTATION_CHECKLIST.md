# WordGauge 详细开发清单

本文档用于梳理当前项目尚未完善或尚未落地的功能，并按优先级拆分为可执行任务。

---

## 一、P0 必须优先修复

这些问题会直接影响测评结果可信度、安全性或线上稳定性，必须优先处理。

### 1. 服务端独立判题
- 状态：未完成
- 问题：
  - 当前后端直接信任前端上传的 `is_correct`。
  - 这意味着用户理论上可以通过篡改请求，让任何题都算“答对”。
  - 会污染正确率、词汇量、IRT 能力值、排行榜、经验值。
- 目标：
  - 后端自己判断用户所选答案是否正确。
- 推荐做法：
  - 方案 A：在 `assessment_sessions` 之外新增一张“题目快照表”，记录每题下发时的 `options` 和 `answer_index`。
  - 方案 B：在后端生成题目时，把题目快照序列化后写进数据库，答题时按 `question_id` 查询快照再判定。
- 涉及文件：
  - `services/api-node/src/main.js`
  - `services/api-node/src/assessment.js`
  - `services/api-node/src/db/schema.js`
- 验收标准：
  - 前端不再上传 `is_correct`。
  - 后端仅依据 `choice_index` 和服务端存储的题目快照判定结果。
  - 篡改请求无法刷分。

### 2. 服务端计时闭环
- 状态：未完成
- 问题：
  - 当前超时结束主要依赖前端倒计时。
  - 用户可以暂停前端、篡改前端逻辑或延迟提交。
- 目标：
  - 后端持有测评开始时间、时限，并在答题接口里强制校验是否超时。
- 推荐做法：
  - 在 `assessment_sessions` 表中增加 `time_limit_sec`、`started_at` 字段。
  - 每次调用 `/answer` 时检查 `Date.now() - started_at` 是否超过时限。
  - 一旦超时，后端直接结束会话并返回最终结果状态。
- 涉及文件：
  - `services/api-node/src/db/schema.js`
  - `services/api-node/src/main.js`
- 验收标准：
  - 即使前端倒计时失效，后端仍能正确判定超时结束。

### 3. 修复 SRS 首次入池时间
- 状态：未完成
- 问题：
  - 当前首次错题入池仍然使用“5 秒后复习”的测试逻辑。
  - 这和正式的艾宾浩斯节奏不一致。
- 目标：
  - 首次入池恢复为正式生产规则，例如 1 天后复习。
- 推荐做法：
  - 将首次入池逻辑与 `/api/srs/review` 中的正式间隔统一管理。
  - 提取一个公共函数，如 `getNextReviewAt(boxLevel)`.
- 涉及文件：
  - `services/api-node/src/main.js`
- 验收标准：
  - 错题进入 SRS 后，首次复习时间为正式规则而不是秒级测试值。

### 4. 管理接口权限隔离
- 状态：未完成
- 问题：
  - 当前 `/api/admin/reload_lexicon` 只要求登录，不区分普通用户和管理员。
- 目标：
  - 引入最基础的管理员权限模型。
- 推荐做法：
  - 在 `users` 表增加 `role` 字段，默认 `user`，管理员为 `admin`。
  - 新增 `requireAdmin` 中间件。
- 涉及文件：
  - `services/api-node/src/db/schema.js`
  - `services/api-node/src/auth.js`
  - `services/api-node/src/main.js`
- 验收标准：
  - 普通用户调用管理接口返回 403。
  - 管理员账号可正常调用。

## 二、P1 产品体验补完

这些功能不一定会导致结果错误，但会明显影响产品完整度和用户体验。

### 5. 匿名测评模式
- 状态：未完成
- 当前情况：
  - 首页写了“匿名测评 + 云端同步（可选）”，但实际上必须登录。
- 目标：
  - 支持未登录用户直接开始测评。
  - 测评结果可临时保存在浏览器，登录后再绑定到账号。
- 推荐做法：
  - 放开 `/setup`、`/quiz`、`/result` 的匿名访问。
  - 未登录时用临时 session 或 localStorage 保存结果。
  - 登录后提供“绑定到账号”入口。
- 涉及文件：
  - `apps/web/src/App.tsx`
  - `apps/web/src/components/ProtectedRoute.tsx`
  - `services/api-node/src/main.js`
- 验收标准：
  - 游客可直接开始测评。
  - 登录后可选择同步历史记录。

### 6. Auth 完整化
- 状态：部分完成
- 已有：
  - 注册、登录、JWT 鉴权、前端退出登录。
- 缺失：
  - Refresh Token
  - 服务端退出登录
  - 更完整的登录过期处理
- 目标：
  - 降低“登录过期后体验很差”的问题。
- 推荐做法：
  - 增加 refresh token 表。
  - 增加 `/api/auth/refresh` 和 `/api/auth/logout`。
- 涉及文件：
  - `services/api-node/src/auth.js`
  - `services/api-node/src/db/schema.js`
  - `apps/web/src/api/client.ts`
- 验收标准：
  - token 过期后可自动刷新。
  - 退出登录后旧 token 不再有效。

### 7. 词库版本化与规则后台化
- 状态：未完成
- 问题：
  - 出题规则和等级映射大多写死在代码里。
  - 后期维护成本高。
- 目标：
  - 让规则可配置、可回滚、可审计。
- 推荐做法：
  - 增加 `rulesets`、`lexicon_versions` 等表。
  - 支持记录启用中的版本号。
- 涉及文件：
  - `services/api-node/src/db/schema.js`
  - `services/api-node/src/main.js`
- 验收标准：
  - 不改代码也能切换规则版本和词库版本。

### 8. 后台管理系统
- 状态：未完成
- 目标：
  - 提供独立后台页面用于：
    - 词库重载
    - 用户统计
    - 测评次数统计
    - 热门错词分析
    - 规则版本切换
- 推荐做法：
  - 新建 `apps/admin`
  - 使用现有 React + Vite 技术栈复用
- 涉及文件：
  - 新增后台前端目录
  - 后端新增 `/api/admin/*`
- 验收标准：
  - 管理员可通过界面完成基础维护，不需要手动改库。

## 八、一句话结论

当前项目已经完成了“可用的词汇测评产品主链路”，但距离“高可信、强运营、可持续扩展的完整产品”还差三块：
- 可信度闭环
- 后台治理能力
- 长期留存机制

下一步最该做的，不是再加花，而是先把“服务端判题”和“服务端计时”补硬。
