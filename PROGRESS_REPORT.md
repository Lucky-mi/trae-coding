# WordGauge 阶段性进展报告 (V1 总结)

## 1. 项目进展 (Current Progress)

截至目前，WordGauge 已从最初的“静态假数据原型”升级为一个**具备前后端全链路闭环、持久化存储、动态出题与数据分析能力的 Web 应用**。核心完成的功能如下：

### 1.1 核心框架与持久化底座
*   **后端架构重构**：从 Python (FastAPI 因环境依赖受阻) 平滑迁移至 `Node.js + Express`，极大提升了本地开发与调试效率。
*   **SQLite 引入与表结构设计**：引入 `better-sqlite3`。实现了 `users` (用户)、`words` (词库)、`assessment_sessions` (测评场次)、`assessment_answers` (答题流水) 四张核心表，完成了无状态到有状态的跨越。
*   **真实词库自动化解析**：实现后端启动时自动读取项目根目录下的 3 份 `.xlsx` 考纲文件（共 3479 词），并通过正则提取、数据清洗后持久化至数据库。

### 1.2 鉴权与安全机制
*   **JWT 用户系统**：基于 `jsonwebtoken` 和 `bcryptjs` 实现了注册、登录、密码哈希存储。
*   **全局路由守卫**：前端引入 `Zustand` 管理用户状态，配合 `ProtectedRoute` 组件拦截未登录访问，实现了页面的安全隔离。

### 1.3 测评引擎核心算法 (V1)
*   **流式自适应出题 (CAT 雏形)**：废弃了一次性下发题目的模式。改为一题一发，根据前一题的 `is_correct` 状态，动态提升或降低下一题的难度级别（Level）。
*   **编辑距离强干扰项 (Levenshtein Distance)**：彻底移除了拼写题兜底逻辑，强制采用“英选汉”单选。干扰项的生成不再是随机抽取，而是通过计算字符串相似度（如 `booth` 与 `boot`）及词性匹配，生成极具迷惑性的选项。

### 1.4 数据中心与互动页面
*   **预估词汇量模型**：基于用户学段与正确率的线性公式，实现了带有绝对数值反馈的“预估词汇量”评分。
*   **全服天梯榜 (Leaderboard)**：实现了按学段分类的全服排行榜，展示用户的最高历史成绩。
*   **词汇中心 (Dictionary & Mistakes)**：包含“全库搜索”与“我的错题本”两大功能。错题本不仅展示正确释义/音标，还统计了该词的历史答错总次数。
*   **可视化报告页 (ResultPage)**：结合 `ECharts` 雷达图与文字分析结论，支持从历史记录页点击回放完整的能力雷达。

## 2. 遇到的问题 (Challenges & Issues)

在 V1 的开发过程中，遇到了以下几个核心技术与业务逻辑卡点：

### 2.1 Python 环境与依赖冲突
*   **现象**：最初计划使用 FastAPI，但在安装 `pydantic-core` 时遭遇当前 Python 环境 (3.14) 的版本不兼容，导致编译失败。
*   **解决**：果断止损，在 5 分钟内将后端整体技术栈切换为 Node.js/Express，确保了项目进度的快速推进。

### 2.2 脏数据解析与表头不一致
*   **现象**：小学/初中词库的英文列名为 `单词`，而高中词库的列名为 `词汇`。最初的脚本正则未覆盖 `词汇`，导致高中阶段的 1865 个单词在解析时被静默跳过，出现“高中题库无法使用”的 Bug。
*   **解决**：增强了 `pickCol` 函数的模糊匹配与别名数组逻辑；同时编写了 `/api/admin/reload_lexicon` 接口，配合临时关闭 SQLite 外键约束，完成了词库的热重载修复。

### 2.3 测评正确率始终为 0% 的 Bug
*   **现象**：重构为流式出题接口后，前端虽然正确计算了选项索引，但未将 `is_correct` 布尔值传回后端，导致后端一律按 `false` 记录答题流水。
*   **解决**：补齐了前端 API 调用的 `is_correct` 字段，并在后端做了基于 `choice_index` 的校验修复。

### 2.4 假难度与测评伪科学（业务痛点）
*   **现象**：提供的 `.xlsx` 数据源中根本没有代表难度的 Level 字段。为了让 CAT 算法运转，只能强行按字母顺序切分“等级”，导致难度梯度毫无教育学依据。
*   **反思**：这也促成了我们后续深度演进计划（V2/V3）中，必须引入 COCA 词频表清洗数据、以及采用真正 IRT 模型的核心动机。

## 3. 经验总结 (Lessons Learned)

1.  **“先跑通，再完美”的敏捷原则**：
    面对 Python 环境阻碍和没有难度等级的数据源时，如果不做变通，项目会一直卡在第 0 步。通过切 Node.js 和“假分桶占位”，我们快速跑通了鉴权、流式出题、错题本的完整链路。地基打好后，再换真实的算法组件（如 IRT/COCA）就变得非常容易。
2.  **数据流向的一致性至关重要**：
    在重构 `/start` 和 `/answer` 接口时漏传 `is_correct` 导致计分失效，暴露出前后端接口在进行流式交互时，状态同步的严谨性要求极高。对于后续开发，所有的 State 变更（尤其是记分、错题）必须有明确且唯一的 Source of Truth。
3.  **核心亮点的非对称优势**：
    普通的背单词软件只是查库展示。而我们在极短时间内引入了 **Levenshtein 编辑距离算法** 生成强干扰项，极大提升了做题的挑战性和区分度。这种用少量代码撬动巨大用户体验的“非对称优势”设计，是后续（比如开发 1v1 进度条追逐）需要继续坚持的思路。
4.  **业务视角的降维打击**：
    仅仅把代码写对是不够的。我们在 ResultPage 加上了“预估词汇量”的绝对数值，以及在顶部加入了“全服天梯榜”，立刻让一个冷冰冰的测评工具带上了**游戏化（Gamification）**与**社交货币**的属性。技术永远是为业务指标（黏性、留存、传播）服务的。
